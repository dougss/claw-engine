# Claw Engine — Design Spec

> Coding agent factory model-agnostic que orquestra sessões de desenvolvimento paralelas,
> roteia tarefas para o modelo mais eficiente, e gerencia o ciclo de vida completo de features.
>
> Criado: 2026-03-31
> Status: Aprovado — pronto para implementação

---

## 1. Visão geral

### O que é

O Claw Engine é um **daemon TypeScript** que funciona como uma fábrica de software automatizada. Ele recebe feature requests, decompõe em DAGs de tasks paralelizáveis, roteia cada task para o modelo mais eficiente (Qwen grátis para o grosso, Claude só quando precisa), e gerencia todo o ciclo — da branch ao PR.

### O que resolve

- **Custo**: R$500/mês no Claude Max com limites imprevisíveis → 70-80% das tasks vão para modelos baratos/grátis
- **Eficiência**: sessões sequenciais → paralelismo real (uma por branch/worktree)
- **Contexto**: CLAUDE.md de ~8k tokens carregado toda sessão → context filtering por task (~2-5k tokens)
- **Visibilidade**: zero controle sobre consumo → dashboard com token tracking, custos, métricas
- **Resiliência**: sessões que morrem por token limit → checkpoint/resume automático

### O que NÃO é

- Não substitui o Claude Code — usa ele como um dos execution engines (Delegate Mode)
- Não é um clone do Claude Code — é um orquestrador que gerencia instâncias de coding agents
- Não requer ML — usa heurísticas simples e estatísticas para routing

### Origem

Arquitetura inspirada no [claw-code](https://github.com/instructkr/claw-code) — reescrita da arquitetura interna do Claude Code exposta em março/2026. Patterns extraídos:

| Pattern do claw-code                                        | Como o Claw Engine usa                               |
| ----------------------------------------------------------- | ---------------------------------------------------- |
| `QueryEnginePort` — token budgeting, rolling window         | Token budget manager com checkpoint em 85%           |
| `bootstrap_graph` — 7 estágios de init                      | Context Builder em 6 camadas otimizadas              |
| `permissions.py` — trust gate                               | Permission system com rules por tool                 |
| `compact.rs` — compactação (preserva 4 msgs, 10k threshold) | Checkpoint/resume com summary + 4 mensagens recentes |
| `tools_snapshot.json` — superfície de ferramentas           | 7 tools essenciais implementadas no harness          |
| `prompt.rs` — system prompt dinâmico                        | System prompt montado em camadas filtradas por task  |

O claw-code é knowledge base estática (não roda em produção). Snapshots JSON são ingeridos no Nexus como skills de referência.

### Dois modos de execução

1. **Engine Mode** — harness próprio para Qwen, DeepSeek, Gemini, qualquer modelo via API. O Claw Engine controla tudo: tools, contexto, permissões, agentic loop.

2. **Delegate Mode** — spawna `claude -p` como subprocesso para tasks complexas. O Claude Code controla a execução, o Claw Engine monitora tokens/saúde e faz checkpoint.

## 3. Agent Harness

O coração do sistema. Executa tarefas de coding usando qualquer modelo com suporte a tool use.

### Agentic Loop

```
User Prompt
     ↓
System Prompt Builder (contexto mínimo + tools disponíveis)
     ↓
┌─────────────────────────────────────┐
│          AGENTIC LOOP               │
│                                     │
│  Prompt → Model API → Response      │
│                ↓                    │
│         Tem tool_use?               │
│          /        \                 │
│        sim        não               │
│         ↓          ↓                │
│   Permission   Resposta final       │
│   Check        → DONE              │
│     ↓                               │
│   Allowed?                          │
│    /    \                           │
│  sim    não                         │
│   ↓      ↓                          │
│  Execute  Deny + feedback           │
│  Tool     ao modelo                 │
│   ↓                                 │
│  Tool Result → volta pro Model      │
│                                     │
│  Max iterations: 16                 │
│  Token budget: configurável         │
└─────────────────────────────────────┘
```

### 7 Tools (MVP)

| Tool         | O que faz                                                          |
| ------------ | ------------------------------------------------------------------ |
| `bash`       | Executa comandos shell (`sh -lc`) com timeout, background, sandbox |
| `read_file`  | Lê arquivo com offset/limit, suporte a imagens e PDFs              |
| `write_file` | Cria arquivo novo (bloqueia overwrite sem leitura prévia)          |
| `edit_file`  | Substituição exata de strings (old_string → new_string)            |
| `glob`       | Busca arquivos por pattern (`**/*.ts`)                             |
| `grep`       | Busca conteúdo em arquivos (regex, context lines, modes)           |
| `ask_user`   | Pergunta ao usuário quando travado (via dashboard/Telegram)        |

### Tool Permissions

```typescript
interface PermissionRule {
  tool: string;
  action: "allow" | "deny" | "prompt";
  conditions?: {
    path_prefix?: string;
    command_pattern?: string;
  };
}
```

**Defaults:**

- `read_file`, `glob`, `grep` → **always allow** (read-only)
- `write_file`, `edit_file` → **allow dentro do workspace da task**
- `bash` → **allow para comandos seguros** (git, npm, node, python, ls), **deny para destrutivos** (rm -rf, drop, force push)
- `ask_user` → **always allow**

### Context Builder (system prompt dinâmico)

Montado em 6 camadas com budget de tokens por camada:

```
┌─────────────────────────────────┐
│ 1. IDENTITY                     │  "You are a coding agent..."
│    (~200 tokens, fixo)          │
├─────────────────────────────────┤
│ 2. TOOLS                        │  Schema das tools disponíveis
│    (~800 tokens, fixo)          │  (só as que a task precisa)
├─────────────────────────────────┤
│ 3. TASK CONTEXT                 │  Descrição da task + posição no DAG
│    (~200-500 tokens, dinâmico)  │
├─────────────────────────────────┤
│ 4. PROJECT CONTEXT              │  CLAUDE.md filtrado por context_filter
│    (~500-2000 tokens, dinâmico) │  Fuzzy match em headings (case-insensitive)
│                                 │  Fallback: primeiras 50 linhas se nenhum match
├─────────────────────────────────┤
│ 5. NEXUS SKILLS                 │  Skills relevantes via semantic search
│    (~300-1000 tokens, dinâmico) │
├─────────────────────────────────┤
│ 6. CHECKPOINT (se resume)       │  Summary da sessão anterior
│    (~500-1000 tokens, opcional) │  + 4 mensagens mais recentes preservadas
└─────────────────────────────────┘

Total estimado: 2000-5500 tokens (vs ~8000+ do Claude Code)
```

### Token Budget Manager

```typescript
interface TokenBudget {
  maxContext: number; // Limite do modelo (128k Qwen, 200k Claude)
  warningAt: number; // 75% → log warning
  checkpointAt: number; // 85% → trigger checkpoint
  reserveForSummary: number; // 10k tokens para summarize

  // Tracking (heurística: len/4 + 1)
  systemPromptTokens: number;
  messagesTokens: number;
  currentTotal: number;
}
```

**Checkpoint flow:**

1. `currentTotal` atinge 85% do `maxContext`
2. Engine injeta: "Resuma o estado atual: o que foi feito, o que falta, próximos passos"
3. Modelo gera summary
4. Engine salva: summary + git diff + 4 mensagens recentes → PostgreSQL
5. Sessão encerrada
6. Nova sessão spawna com summary como camada 6 do Context Builder

## 4. Decomposer + DAG

Recebe feature requests e gera DAGs de tasks executáveis.

### Flow

```
Feature Request (texto livre)
     ↓
Repo Discovery (se repo não tem CLAUDE.md)
  → lê tree -L 2, package.json, README.md
  → gera mini-contexto (~500 tokens)
     ↓
Nexus Query (skills + contexto relevantes)
     ↓
Model (Qwen) gera DAG JSON
     ↓
Validação (Zod schema)
     ↓
DAG registrado no PostgreSQL
```

### DAG Schema

```typescript
interface WorkItemDAG {
  title: string;
  tasks: TaskNode[];
  edges: DependencyEdge[];
}

interface TaskNode {
  id: string; // "task-1"
  repo: string;
  branch: string;
  description: string;
  complexity: "simple" | "medium" | "complex";
  context_filter: string[]; // Seções do CLAUDE.md a injetar
  nexus_skills: string[]; // Skills do Nexus a injetar
  mcp_servers: string[]; // MCPs a injetar (vazio = todos)
  estimated_tokens: number;
  retry_policy: RetryPolicy;
}

interface DependencyEdge {
  from: string;
  to: string;
  type: "blocks" | "informs"; // blocks = hard dep, informs = soft (passa context)
}

interface RetryPolicy {
  max_attempts: number; // default: 3
  backoff: "linear" | "exponential";
  escalate_model_on_retry: boolean;
  on_failure: "block_dependents" | "skip_and_continue" | "cancel_work_item";
}
```

### Error Recovery no DAG

```
Task FAILED
     ↓
  attempt < max_attempts?
    SIM → retry com backoff
      → escalate_model? → próximo modelo na fallback chain
      → re-queue no BullMQ
    NÃO → on_failure:
      'block_dependents' → dependentes ficam BLOCKED, notifica user
      'skip_and_continue' → marca SKIPPED, dependentes rodam com aviso
      'cancel_work_item' → cancela TUDO, cleanup branches + worktrees
```

**Escalation storm prevention:** antes de escalar modelo, compara error messages. Se mesmo tipo de erro em múltiplos modelos → `NEEDS_HUMAN_REVIEW` (problema é o input, não o modelo).

```typescript
function shouldEscalate(attempts: AttemptLog[]): boolean {
  const errors = attempts.map((a) => classifyError(a.error));
  const allSameClass = new Set(errors).size === 1;
  if (allSameClass) return false;
  return true;
}
```

**Defaults por complexidade:**

| Complexidade | max_attempts | escalate | on_failure        |
| ------------ | ------------ | -------- | ----------------- |
| simple       | 3            | true     | skip_and_continue |
| medium       | 2            | true     | block_dependents  |
| complex      | 2            | false    | block_dependents  |

## 5. Router

Recebe cada task e decide qual modelo executa e em qual modo.

### Decisão em 3 camadas

```
Task do Scheduler
     ↓
  CAMADA 1: Regras estáticas
    complexity == 'complex' → Claude (Delegate Mode)
    complexity == 'simple'  → Qwen (Engine Mode)
    complexity == 'medium'  → Camada 2
     ↓
  CAMADA 2: Keyword scoring
    Score > 0 → Claude | Score <= 0 → Qwen
     ↓
  CAMADA 3: Budget check
    Claude Max perto do limite? → força Qwen + aviso
```

### Complexity Signals (configurável)

```typescript

  investigate: +2,
  architecture: +3,
  "cross-repo": +4,
  migration: +2,
  security: +2,
  crud: -2,
  boilerplate: -3,
  test: -1,
  rename: -2,
  "add field": -2,
  "create endpoint": -1,
};
```

### Fallback Chain

```typescript
const DEFAULT_CHAIN: ModelTier[] = [
```

## 6. Session Manager

Orquestra o lifecycle de cada sessão individual.

### Lifecycle

```
PENDING
  ↓
MERGING_DEPENDENCY (se depende de task concluída)
  │  git merge da branch da dependência no worktree
  │  Conflito? → modelo resolve ou NEEDS_HUMAN_REVIEW
  ↓
PROVISIONING
  │  Criar workspace (git worktree)
  │  npm ci (sem symlinks de node_modules)
  │  Montar system prompt (Context Builder)
  │  Resolver modelo (Router)
  ↓
STARTING
  │  Engine Mode: conectar à API do modelo
  │  Delegate Mode: spawnar processo claude -p
  ↓
RUNNING
  │  Agentic loop executando
  │  Health checks a cada 30s
  │  Token tracking contínuo
  │  Stream events → telemetry + dashboard
  ↓
  ├→ CHECKPOINTING (token limit 85% ou stall)
  │    Summarize → salva state → mata sessão
  │    → RESUMING → nova sessão com summary → RUNNING
  │
  ├→ VALIDATING (sessão completou)
  │    typecheck, lint, test (por stack)
  │    → PASSED → COMPLETED
  │    → FAILED → re-inject errors → RUNNING (max 2 retries validação)
  │    → ainda falha → NEEDS_HUMAN_REVIEW
  │
  ├→ STALLED (timeout sem output)
  │    Engine: 60s | Delegate: 300s
  │    → retry? → STARTING
  │    → max retries? → FAILED
  │
  └→ FAILED
       → Router: retry, escalate, ou NEEDS_HUMAN_REVIEW
       → Cleanup workspace
```

### Workspace Isolation (git worktree)

```typescript
interface Workspace {
  taskId: string;
  repoPath: string; // repo original
  worktreePath: string; // ~/server/.worktrees/<task-id>/
  branch: string;
  createdAt: Date;
}
```

```bash
# Provisioning
git -C ~/server/apps/finno worktree add \
  ~/server/.worktrees/task-abc123 \
  -b feat/oauth-provider

# Instalar dependências (npm ci, sem symlinks)
cd ~/server/.worktrees/task-abc123 && npm ci

# Cleanup após PR
git -C ~/server/apps/finno worktree remove \
  ~/server/.worktrees/task-abc123
```

### Git Coordination entre tasks dependentes

```
task-1 completa (branch feat/oauth-provider)
  ↓
task-2 entra em MERGING_DEPENDENCY
  ↓
git -C <worktree-task-2> merge feat/oauth-provider
  ↓
Conflito?
  NÃO → task-2 continua normalmente
  SIM → modelo recebe conflitos no contexto para resolver
      → se falhar → NEEDS_HUMAN_REVIEW
```

### Startup Reconciliation

Roda automaticamente no boot do daemon:

```typescript
async function reconcileOnStartup(): Promise<void> {
  // 1. Listar worktrees existentes em disco
  const diskWorktrees = await listWorktrees();

  // 2. Comparar com tasks ativas no PostgreSQL
  const activeTasks = await getActiveTasks();

  // 3. Worktrees sem task ativa = órfãos → cleanup
  for (const wt of diskWorktrees) {
    if (!activeTasks.has(wt.taskId)) {
      await cleanupOrphanWorktree(wt);
    }
  }

  // 4. Tasks RUNNING quando crashou → INTERRUPTED → re-queue
  for (const task of activeTasks) {
    if (task.status === "running") {
      await markInterrupted(task);
    }
  }
}
```

## 14. MCP Integration (tools extensíveis)

O harness começa com 7 tools built-in. MCP (Model Context Protocol) torna o sistema **infinitamente extensível** — qualquer MCP server configurado vira automaticamente tools disponíveis para os agents, sem alterar código do harness.

### Arquitetura

```
┌─────────────────────────────────────────────┐
│            Tool Registry                     │
│                                              │
│  Built-in (7):                              │
│    bash, read_file, write_file, edit_file,  │
│    glob, grep, ask_user                     │
│                                              │
│  MCP Auto-discovered:                       │
│    nexus → nexus, nexus_save               │
│    supabase → query, insert, migrate       │
│    github → create_issue, create_pr        │
│    slack → send_message                    │
│    <qualquer-mcp> → suas tools             │
└─────────────────────────────────────────────┘
```

### Separação por modo

- **Delegate Mode** (`claude -p`) — herda MCPs automaticamente. Claude Code já lê seus MCPs de `~/.claude/settings.json`. O Claw Engine não precisa fazer nada.
- **Engine Mode** (Qwen/DeepSeek) — usa config do `config.yaml`, com opção de herdar do Claude Code como base:

```yaml
# config.yaml
mcp:
  inherit_from: "~/.claude/settings.json" # herda MCPs do Claude Code como base
  servers:
    # Overrides e adições
    nexus:
      command: "node"
      args: ["/Users/macmini/server/apps/nexus/server/dist/mcp-stdio.js"]
      env:
        DATABASE_URL: "postgres://nexus:nexus_local@127.0.0.1:5432/nexus"
      permissions: "allow" # read-only, safe
    supabase:
      command: "npx"
      args: ["supabase-mcp"]
      permissions: "prompt" # modifica dados, gate necessário
    github:
      command: "gh-mcp"
      permissions:
        create_issue: "allow"
        create_pr: "allow"
        delete_repo: "deny" # nunca
```

### MCP Tool Registry

```typescript
interface MCPToolRegistry {
  // Boot: conecta em todos os MCP servers configurados
  // Descobre tools via MCP protocol (tools/list)
  // Registra no Tool Registry junto com as 7 built-in
  discover(): Promise<MCPTool[]>;

  // Por task: filtra quais MCPs são relevantes
  filterForTask(task: TaskNode): MCPTool[];

  // Executa tool call via MCP protocol (tools/call)
  execute(
    serverName: string,
    toolName: string,
    args: unknown,
  ): Promise<ToolResult>;
}
```

### Filtragem por task

O `TaskNode` ganha campo `mcp_servers` — o Decomposer seleciona quais MCPs cada task precisa:

```typescript
interface TaskNode {
  // ... existing fields ...
  context_filter: string[]; // seções do CLAUDE.md
  nexus_skills: string[]; // skills do Nexus
  mcp_servers: string[]; // MCPs a injetar (NOVO)
  // Se vazio → usa todos os configurados
  // Se ['nexus', 'supabase'] → só esses
}
```

**Por que filtrar:** cada MCP server expõe N tools com JSON schemas. 5 MCP servers com 30 tools = ~3000-5000 tokens só de schemas. Isso come o budget economizado no Context Builder. O Decomposer seleciona, o Context Builder injeta só os schemas relevantes.

### Permission system para MCP tools

MCP tools passam pelo mesmo permission system das built-in:

```typescript
// Permissions podem ser definidas por:
// 1. Server inteiro: permissions: "allow" | "deny" | "prompt"
// 2. Tool individual: permissions: { tool_name: "allow" | "deny" }
// 3. Default: "prompt" (se não configurado)
```

### Implementação (MCP client)

O Engine Mode precisa de um MCP client para conectar nos servers. O `@modelcontextprotocol/sdk` (já usado pelo Nexus, v1.12) tem o client pronto:

```typescript
import { Client } from "@modelcontextprotocol/sdk/client/index.js";
import { StdioClientTransport } from "@modelcontextprotocol/sdk/client/stdio.js";

// Boot: conectar em cada MCP server
const transport = new StdioClientTransport({ command: "node", args: [...] });
const client = new Client({ name: "claw-engine" });
await client.connect(transport);

// Discover
const { tools } = await client.listTools();

// Execute (quando modelo pede tool call de um MCP)
const result = await client.callTool({ name: "nexus", arguments: { ... } });
```

### Tool schema translation

Modelos diferentes esperam tool schemas em formatos diferentes. O harness precisa traduzir:

```
MCP tool schema (JSON Schema)
     ↓
  Anthropic format (tool_use blocks) → para Claude API
  OpenAI format (function calling) → para Qwen/DeepSeek via DashScope
```

O `ModelAdapter` já abstrai o provider — a tradução de schema é responsabilidade de cada adapter.

### Impacto nas fases

| Fase                     | Mudança                                                                       |
| ------------------------ | ----------------------------------------------------------------------------- |
| Fase 1 (Harness)         | 7 built-in tools. Nenhuma mudança.                                            |
| Fase 2 (Session Manager) | `claw run` funciona com built-in tools.                                       |
| Fase 3 (Decomposer)      | `TaskNode` ganha campo `mcp_servers: string[]`.                               |
| Fase 5 (Integrations)    | **MCP client no Engine Mode.** Tool discovery. Permission rules. Config YAML. |
| Fase 6 (Dashboard)       | Dashboard mostra MCP tools disponíveis por sessão.                            |

MCP não bloqueia as fases iniciais. O harness funciona com 7 tools no começo, MCP é adicionado na Fase 5 quando a base está sólida.

## 15. Riscos e mitigações

| Risco                                  | Impacto                                 | Mitigação                                           |
| -------------------------------------- | --------------------------------------- | --------------------------------------------------- |
| Qwen não dá conta de tarefas "simples" | Split real fica 40/60 ao invés de 70/30 | Fallback chain automática + learning loop ajusta    |
| Mac Mini 16GB RAM                      | OOM com 3 sessões + Docker              | max_parallel_delegate: 1, monitorar memory_pressure |
| 256GB SSD                              | Worktrees + node_modules enche          | Cleanup agressivo, monitorar disk, alertas em 20GB  |
| Claude Code muda CLI flags             | Delegate Mode quebra                    | Abstração fina no ClaudePipeAdapter                 |
| Rate limits Bailian                    | Sessões paralelas bloqueiam             | Token bucket por provider no BullMQ                 |
| Escalation storm                       | Gasta tokens sem resolver               | classifyError + NEEDS_HUMAN_REVIEW                  |
| Manutenção solo                        | Dev solo mantendo sistema distribuído   | Manter simples, alertas automáticos, bons testes    |
